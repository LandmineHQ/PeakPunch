# BuddyClimb Project Notes

## Project Overview

- This repository contains PEAK BepInEx mods rebuilt from the PEAKModding BepInEx template.
- Gameplay mod name: `BuddyClimb`.
- Gameplay plugin GUID and assembly name: `com.github.LandmineHQ.BuddyClimb`.
- Tooling mod name: `PeakDummyTools`.
- Tooling plugin GUID and assembly name: `com.github.LandmineHQ.PeakDummyTools`.
- Experimental optimization mod name: `PeakPlayerLOD`.
- Optimization plugin GUID and assembly name: `com.github.LandmineHQ.PeakPlayerLOD`.
- Experimental route planning mod name: `PeakRoutePlanner`.
- Route planning plugin GUID and assembly name: `com.github.LandmineHQ.PeakRoutePlanner`.
- Target framework: `netstandard2.1`.
- Main solution: `BuddyClimb.slnx`.
- Gameplay project: `src/BuddyClimb/BuddyClimb.csproj`.
- Tooling project: `src/PeakDummyTools/PeakDummyTools.csproj`.
- Optimization project: `src/PeakPlayerLOD/PeakPlayerLOD.csproj`.
- Route planning project: `src/PeakRoutePlanner/PeakRoutePlanner.csproj`.
- These projects are independent mods. Do not add project references or BepInEx dependencies between them unless the user explicitly changes that boundary.
- Runtime dependency target is PEAK's BepInExPack, currently `BepInEx-BepInExPack_PEAK` `5.4.75301`.

## Build And Validation

- `Config.Build.user.props` is the local machine configuration file for this template, similar in role to a `.env` file. It is gitignored and should not be committed.
- Copy `Config.Build.user.props.template` to `Config.Build.user.props` and set `PEAKGameRootDir` there when the default Steam install path is not correct for the current machine.
- Use `scripts/build.ps1` for local validation. The script requires `Config.Build.user.props` and relies on its MSBuild properties instead of shell environment variables.
- Default validation: `.\scripts\build.ps1` runs `release`.
- Debug validation: `.\scripts\build.ps1 debug`
- Release packaging validation: `.\scripts\build.ps1 release`
- Thunderstore publish validation/push: `.\scripts\build.ps1 push`. This runs Release with `-p:PublishTS=true` and requires the local Thunderstore token setup from `Config.Build.user.props`.
- `push` mode runs in the current PowerShell process and does not request UAC elevation. Thunderstore publish failures should be read from the same terminal output.
- Keep Thunderstore token lookup commands in `Config.Build.user.props.template` on a single XML text line; MSBuild `Exec` preserves embedded newlines and can fail with exit code `9009` if the command is split.
- The build script passes verbosity as `dotnet build -v <Verbosity>` and defaults to `minimal`; use `-Verbosity d` to match the PEAKModding packaging docs' detailed output. It does not override `DeployModFiles`; set deployment behavior in `Config.Build.user.props`.
- Normal solution builds should produce four DLLs: `com.github.LandmineHQ.BuddyClimb.dll`, `com.github.LandmineHQ.PeakDummyTools.dll`, `com.github.LandmineHQ.PeakPlayerLOD.dll`, and `com.github.LandmineHQ.PeakRoutePlanner.dll`.
- Release package output is under `artifacts/thunderstore/release/`. `BuddyClimb` is Thunderstore-packable; `PeakDummyTools`, `PeakPlayerLOD`, and `PeakRoutePlanner` are local DLLs and are not packed unless their project metadata is intentionally changed.
- `artifacts/` is build output and should not be committed.
- When reading text files from PowerShell, use explicit UTF-8, for example `Get-Content -Encoding UTF8 <path>`.
- When checking PEAK internals, resolve the PEAK Managed directory from `Config.Build.user.props`, template defaults, or a discovered local Steam library; then inspect `PEAK_Data/Managed/Assembly-CSharp.dll`.
- If Codex has ILSpy tools exposed, use them directly. Otherwise, search likely VS Code extension directories for an `icsharpcode.ilspy-vscode-*` installation and use its bundled backend assemblies (`Mono.Cecil` / `ICSharpCode.Decompiler`) for metadata or decompilation before falling back to rough guesses.

## Implementation Notes

- Harmony comes from the template's NuGet dependency (`HarmonyX` `2.9.0`). Do not copy Harmony assemblies from old build outputs, and do not upgrade HarmonyX unless there is a concrete compatibility reason.
- The primary gameplay hook is `CharacterInteractiblePatch`.
- `BuddyClimbConfig` owns gameplay config entries. `EnableBackpackTransfer` defaults to false; when false, carrier backpacks still block BuddyClimb interactions.
- `BuddyClimbConfig` runtime config hot reload uses `FileSystemWatcher` callbacks with a debounced timer to call this mod's `Config.Reload()`. Config edits apply without restarting the game. Do not add `Plugin.Update()` dirty polling or fixed-interval file polling.
- The climb prompt must go through `BuddyClimbLocalization`; do not hardcode player-facing prompt text in patches.
- Use a distinct `BuddyClimbTextKey` for each full prompt. Do not build player-facing prompt variants by concatenating localized fragments.
- Current localization supports English plus Simplified/Traditional Chinese, with unsupported languages falling back to English.
- Conscious local players carried through BuddyClimb can press Space to request a drop through the carrier's `CharacterCarrying.Drop` RPC. This Space handling must consume/suppress the same-frame jump input and must not activate for vanilla unconscious carry states.
- BuddyClimb-owned carried players should use PEAK's spectator camera locked to `Character.localCharacter`, so the carried player watches themselves while riding. Do not override PEAK's default ghost spectator zoom range. This camera patch is disabled when the `nakazora.peak.piggyback` plugin is detected, because Piggyback implements its own spectator behavior.
- BuddyClimb interaction stacking allows a player currently being carried to be climbed, but targets that are already carrying someone remain blocked.
- BuddyClimb does not redirect climb interactions from a carrier to that carrier's `data.carriedPlayer`. All carried players get a trigger interaction proxy because PEAK disables their ragdoll colliders through `ToggleCarryPhysics(true)`, so normal interaction raycasts otherwise cannot hit them directly. This proxy is not limited to BuddyClimb-owned carry states.
- PEAK's vanilla `E` drop prompt and `Interact` drop path must stay suppressed for BuddyClimb-carried targets; BuddyClimb drop remains the carried local player's Space action.
- Local players may start BuddyClimb while already carrying someone, but `StartCarry` must reject cyclic carry links where the proposed carrier is already carried by the proposed carried player.
- Local or currently controlled players may not start BuddyClimb while holding an item; keep that restriction in `CanClimb`. A target may still be climbed while that target is holding an item; do not make `CanBeClimbed` reject the target solely because `character.data.currentItem` is set.
- When `EnableBackpackTransfer` is true and the carrier has a backpack, move the carrier's backpack to the carried player during BuddyClimb carry start. If the carried player already has a backpack, create the dropped backpack from a snapshot of the carried player's backpack slot, immediately clear that slot, then move the carrier's backpack in the same interaction. Do not restore the transferred backpack to the carrier on drop.
- BuddyClimb climb interactions must not call the vanilla `RPCA_PassOut`; carried players should remain conscious. Send `RPCA_StartCarry` directly from the carrier's `PhotonView`, and let `CharacterCarryingPatch` apply `isCarried`, `carrier`, and `carriedPlayer`. Avoid `CharacterCarrying.StartCarry()` when the direct RPC path is needed because PEAK initializes its private `character` field in `Start()`.
- `PeakDummyTools` settings use BepInEx `Config.Bind` in `PeakDummyToolsConfig`.
- `PeakDummyTools` runtime config hot reload uses `FileSystemWatcher` callbacks with a debounced timer to call that mod's `Config.Reload()`. Do not add `Plugin.Update()` dirty polling or fixed-interval file polling.
- Dummy player spawning is client-owned and requires an active Photon room. It creates a marked test `Character` at the local player's center position; the `Character.Awake` patch must temporarily mark this instance as a bot before PEAK's original player registration runs, or the client's real `Character` can be disabled by `PlayerHandler.RegisterCharacter`.
- Dummy characters cannot be real Photon room players. Keep their identity isolated with `PeakDummyTools`' synthetic local `Player` mapping, `Character.player` / `Player.character` / `NetworkingUtilities` patches, and UI name handling so they never resolve to the local player.
- Dummy names are generated through `PeakDummyToolsLocalization` and should remain incrementing and localized. They include the spawning client's cached local player name as a prefix, for example `PlayerName[假人1]`. Do not rely on `gameObject.name` alone, because PEAK resets names in `Character.Start`.
- Dummy spawning defaults to `LeftAlt+G`. Dummy control switching defaults to `LeftAlt+T`, is client-owned, and switches `Character.localCharacter`, local `Player.character`, the local actor's `PlayerHandler` character lookup, and the local Photon Voice recorder to any targeted non-current `Character`, including other players, generated dummies, and bots, without moving either character. Pressing the switch shortcut while controlling another character and targeting the original local character, or with no switch target, returns control to the original local character. Dummy deletion defaults to `LeftAlt+D` and deletes only dummies generated by this client through Photon destruction; if no dummy is targeted, delete the next still-existing locally generated dummy from the spawn queue.
- Dummy control switching should show its own `Alt + T` interaction prompt row from the UI layer, and dummy deletion should show its own `Alt + D` row below it when the current hovered character is a deletable locally generated dummy. Do not append this text through `CharacterInteractible.GetInteractionText`, because that reuses PEAK's normal `E` interaction slot. Keep these rows attached to PEAK's native `GUIManager.RefreshInteractablePrompt` flow, populate them only from the current hovered `CharacterInteractible`, and clone native prompt rows for the custom lines instead of using an independent overlay canvas. DummyTools may contribute its own `CharacterInteractible.IsInteractible` truth for switchable or deletable targets so custom hover/prompt rows do not depend on PEAK's primary `E` or secondary right-click actions being available. If the same hovered target remains interactible but its primary/secondary/custom prompt state changes, trigger the native refresh from a state-change check instead of writing PEAK's primary or secondary UI directly.
- `PeakPlayerLOD` owns experimental player visual LOD/model optimization. Keep it isolated from BuddyClimb and PeakDummyTools so crashes or visual regressions can be disabled by removing only that DLL.
- `PeakPlayerLOD` runtime config hot reload uses `FileSystemWatcher` callbacks with a debounced timer to call that mod's `Config.Reload()`. Do not add `Plugin.Update()` dirty polling or fixed-interval file polling.
- `PeakPlayerLOD` should avoid disabling root `Character`, `PhotonView`, colliders, animation/IK components, or interaction components. Prefer renderer-only experiments plus explicit cleanup that restores renderer state.
- `PeakPlayerLOD` keeps the nearest configured non-local players on their original renderers and represents farther non-local players with a skin-only renderer proxy on the original character: keep the base skin/body and face expression renderers, hide clothing, hats, accessories, shadows, chicken, skeleton, and other cosmetic renderers. Do not create separate mesh proxy objects for this mode.
- `PeakPlayerLOD` LOD transitions use `PlayerVisualLodSwitchDebounceSeconds` so players near the full-detail boundary do not rapidly toggle renderer states.
- `PeakRoutePlanner` owns experimental surface route planning. Keep it isolated from BuddyClimb, PeakDummyTools, and PeakPlayerLOD. It should not auto-control players in its first version; it computes and renders routes only.
- `PeakRoutePlanner` must keep Unity Physics queries on the Unity main thread. Background work may only process pure data snapshots such as sampled `SurfacePoint` and validated `RouteEdge` values.
- `PeakRoutePlanner` should plan over exterior collision surfaces, not direct 3D straight lines. Capture the local player's position when the route key is pressed, build an initial guide by adaptively projecting the player-to-campfire line onto valid terrain surfaces unless manual target config is enabled, then sample progressively with a bidirectional surface frontier grid from both the player point and campfire point, validate movement edges, build a target-rooted surface distance field, and run stamina-aware A* over validated directed edges.
- `PeakRoutePlanner` route planning defaults to `LeftAlt+Comma`; clearing/canceling the route defaults to `LeftAlt+Period`. Intermediate route previews should update throttled `LineRenderer` output instead of waiting only for the final route result; the start-side validated route preview and target-side frontier preview must use separate line renderers so disconnected graph components are never visually joined by a fake straight segment. The active sampling area should be shown with one reusable translucent white cube created when planning starts, updated as each sampling window changes, and destroyed only when the full planning run is finished or cleared; it is diagnostic only and must not be treated as a validated route edge.
- `PeakRoutePlanner` main-thread work must stay hard-budgeted. Sampling should generate surface-guide corridor ray origins incrementally, candidate edge construction must be split across frames, point/candidate caps should prevent one corridor attempt from locking the Unity game thread, and background path-search state caps should keep A* from monopolizing CPU.
- `PeakRoutePlanner` must cache sampled surface points, processed ray keys, generated candidate keys, and validated directed edges across expansion attempts in the same route-planning run. Expanding the search should add only newly uncovered samples/edges and keep the last visible preview instead of clearing back to a two-anchor graph. Frontier expansion should use coarse cached seed cells while keeping ray/sample spacing fine, so flat ground does not exhaust the point cap before the route can advance.
- `PeakRoutePlanner` long routes must use a committed forward-window model instead of one ever-growing global graph. Once a partial route advances far enough toward the target, append that segment to the rendered committed path, reset the local sampler/edge graph, and continue from the new frontier. Keep bidirectional frontier sampling enabled by default within the current bounded window so the campfire/target side can build validated edges and a target-rooted distance field before the two sides connect. Do not widen the same local corridor by default when a local route stalls; route recovery should prefer committing a short validated detour segment and moving the sampling window to another reachable frontier. Partial search should first choose a reachable state whose reconstructed path meets `MinimumPartialSegmentDistance`; sub-threshold partials are only a preview fallback. Corridor expansion is an opt-in diagnostic fallback only.
- `PeakRoutePlanner` movement costs should be derived from PEAK runtime movement components when planning starts: use `CharacterMovement` jump/sprint fields, `CharacterClimbing` climb speed/stamina fields, and `Ascents.climbStaminaMultiplier` instead of hardcoded jump distance or climb stamina guesses. Full-stamina climb distance can be computed from climb speed and climb stamina drain; jump and air-transfer distances are conservative estimates from PEAK fields/IL behavior, not exact replay of the player controller, and must be clamped by the configured maximum jump/air-transfer distances so heuristic estimates do not create 10m+ shortcut edges. If route search fails, keep the last guide/intermediate line visible so failure is distinguishable from a successful final path swap.
- `PeakRoutePlanner` route edges are directional. Do not turn a valid downward walk/drop/jump into a valid upward shortcut by automatically adding reverse graph edges. Sampling should also filter stacked vertical ray hits so a tree/canopy/overhang above a valid ground hit is not treated as the default route layer. Edge validation must reject mid-segment physics hits for airborne `StandJump`/`AirTransfer` chords, while surface-following `StandWalk`/`SurfaceClimb` edges should be validated by sampled surface support/clearance instead of a single horizontal spherecast that can treat the ground surface as a blocking collision.
- `PeakRoutePlanner` must discover climbable wall surfaces with budgeted horizontal/normal probes in addition to Foothold-style downward terrain rays. `SurfaceClimb` edges must stay short and surface-continuous: validate outside-wall clearance, intermediate support along the expected surface normal, and compatible neighboring normals so the route cannot connect through the inside or opposite side of the same rock/wall.
- `PeakRoutePlanner` dense standable regions should build edge candidates from a local walk-sized spatial grid only. Do not let every flat-ground point search the full stand-jump radius; reserve the larger candidate grid for climbable points, sparse standable boundary points, and actual gap-solving jump/air-transfer candidates.
- `PeakRoutePlanner` edge candidate construction must yield after bounded work even when a sampled point generates no valid candidate. Avoid inner loops that scan many old/cached points inside one `ProcessFrame` unit, because large flat regions can otherwise stall the Unity main thread despite frame budgets.
- `PeakRoutePlanner` A* movement cost should be driven by validated edge distance, a target-rooted surface distance field, stamina buckets, and progress along the surface guide, not by raw 3D distance to the target. The target distance field may collapse validated `StandWalk`-connected standable points into region units while keeping final rendered paths on validated directed edges. Penalize repeated backtracking, long world-space steps with weak guide progress, and expensive move kinds so normal walk/climb edges dominate and `StandJump`/`AirTransfer` are rare gap-solving edges, but do not hard-filter short-term detours that move away from the guide when the sampled graph shows no direct forward route.
- `PeakRoutePlanner` may remap the exact player/campfire anchor points to nearby connected surface graph nodes when the exact anchors have no validated outgoing/incoming route edges. Log the remapped indices so in-game failures can be diagnosed from `startOutgoing`, `targetIncoming`, `effectiveStart`, `effectiveTarget`, and reachable-node counts.
- `PeakRoutePlanner` surface sampling should run as a bidirectional frontier grid from both A (campfire/target) and B (captured player point). Each expanded frontier seed should use the Foothold-style `SurfaceSamplingWindowRadius` as an XZ radius (default 10m, 0.5m grid spacing), but sampling should not pre-fill the whole cube with rays. Start from the accepted seed point and flood-fill nearby surface samples by probing 0.5m neighbors from each confirmed surface point; each neighbor is projected from that point's surface normal/tangent, then newly confirmed points continue the same local expansion. This keeps cave sampling on the cave floor/wall layer, lets climbable wall points continue upward through surface-normal/tangent probes, and avoids jumping to a mountain roof or unrelated internal collider top above the seed layer. When a 0.5m standable neighbor probe fails at a boundary, enqueue a small, capped set of gap landing probes up to `MaxStandJumpDistance` to discover disconnected opposite-side standable samples; these gap probes only add sampled points, and `EdgeValidator` must still validate the eventual `StandJump`/`AirTransfer` edge before any rendered route can cross the gap. Vertical downward rays are fallback/anchor-style local-layer checks only. Classify points with the Foothold-style standable-angle test, and enqueue nearby standable/climbable surface points for later expansion toward the opposite side. The diagnostic sampling cube should use the Foothold-style volume, with XZ size `radius * 2` and Y size `radius * 3`, while the actual rays inside it are sparse neighbor probes. One local planning attempt should expand only `MaxSamplingWindowsPerSide` windows per side (default 1); if more windows are enabled for diagnostics, cube centers must come from accepted exterior surface seeds and pass a cube-diameter XZ AABB non-overlap check rather than being synthesized as blind grid centers. Because Foothold-style vertical rays can see stacked, internal, or lower surfaces, guide projection and sampling must prefer the current or previous height-continuous surface layer instead of the lowest ray hit; starting on rocks or upper ledges must not project to lower ground underneath. When a midpoint is not standable, search nearby for a standable replacement on the same height-continuous layer before falling back to climbable surface points. Partial previews should prefer the path from the start-reachable distance field to the reachable node closest to the campfire/target, use binary guide-progress target selection that converges to `AdaptiveGuideMinimumStep` (default 0.5m) as a fallback, prefer low lateral guide distance, penalize recently committed frontier positions to avoid local loops, mark stalled non-progress frontiers as blocked, treat partial routes whose net guide progress is below `MinimumFrontierAdvanceDistance` as blocked dead ends, use recovery detours to backtrack or sidestep away from blocked frontiers, and only fall back to a short validated local preview when the current validated subgraph has no forward guide-progress node.
- `PeakRoutePlanner` must treat ordinary top-down exterior sampling like a depth-buffer/z-buffer query, but local route windows inside caves or under overhangs should choose the valid hit closest to the seed height layer instead of blindly accepting the first hit from above. Standing capsule clearance remains a secondary validation for accepted standable hits, and must include an upward headroom probe so cave ceilings or mountain interiors in the same mesh collider can still reject a false standable top.
- `PeakRoutePlanner` surface sampling should use a window-level broadphase where possible: each expanded frontier window may collect nearby terrain colliders with `OverlapBoxNonAlloc`, then resolve surface-projection and wall probes with per-collider `Collider.Raycast`. Anchor/fallback vertical rays should use `Physics.RaycastNonAlloc` over the seed-local height span so multi-hit cave or overhang cases can select the correct height-continuous layer instead of being truncated to a collider's first hit. Fall back to global casts for directed probes when the window collider set is unavailable or too large, and log broadphase window/fallback counts for diagnosis.
- `PeakRoutePlanner` may suppress redundant flat standable samples by keeping only a few representative points per coarse flat-region cell. Do not suppress forced anchor points, non-flat standable points, climbable points, wall probes, or boundary-like samples needed for route transitions.
- `PeakRoutePlanner` enforces a runtime floor of 192 `MaxGuideCurveSamples` even if an older config file still has the previous lower default, because the adaptive 0.5m guide step needs enough centerline samples to be useful on normal PEAK route distances.
- `PeakRoutePlanner` enforces runtime floors for `MaxSurfacePointsPerAttempt` and `MaxEdgeCandidatesPerAttempt` so old config files do not cap the cached graph at the earlier small debug values and prevent later corridor expansions from adding useful samples.
- `PeakRoutePlanner` intermediate previews must not connect unvalidated sampled surface points or the raw surface guide as if they were a real route. Render previews only through validated directed edges using the same guide-progress-biased A* style search; if no complete route is available yet, a partial preview may render only the validated edge path from the start to the best reachable progress node, and a separate target-side preview may render validated directed edges that can reach the campfire/target. Final route rendering should be accepted only when the returned node path is still composed of the current validated directed edges.
- `PeakRoutePlanner` edge validation should prioritize short, ordinary edges near the current frontier so intermediate previews become visible quickly while the rest of the candidate graph continues validating under the frame budget. Runtime intermediate preview rendering should cap old high throttle config values so diagnostics update responsively.

## Compatibility Boundaries

- Keep the plugin GUID stable unless intentionally creating a new incompatible package identity.
- Keep Thunderstore metadata in MSBuild/ThunderPipe configuration; do not reintroduce the old `thunderstore.toml` workflow.
- Prefer PEAK's current `Assembly-CSharp.dll` metadata from the resolved local Managed directory when checking API compatibility.
- Before changing BuddyClimb interaction behavior, verify normal remote player targets. When changing PeakDummyTools, verify spawned dummies still behave like generic non-local player characters. When changing PeakPlayerLOD, verify original renderers are restored when the feature or plugin is disabled. When changing PeakRoutePlanner, verify physics-query budgets still keep sampling/edge validation split across frames and that route edges do not intentionally use unvalidated through-wall shortcuts.
